# CreatorStack AI Release Strategy

CreatorStack AI uses a two-branch release model:

- `develop` is the integration branch for staging builds.
- `main` is the production-ready branch.
- Feature branches are created from `develop` and merged back into `develop` by pull request.
- Production releases are created only from `main` by pushing a version tag such as `v1.1.0`.

## Feature And Staging Flow

1. Create feature branches from `develop`.
2. Open pull requests into `develop`.
3. Merge only after the CI workflow passes.
4. Every push to `develop` runs the staging package workflow.
5. The staging workflow uploads a `creatorstack-ai-staging-<short_sha>.zip` artifact for manual installation in WordPress or WordPress Playground.

The staging artifact uses a package-only version in the form `0.0.0-develop.<run_number>`. This version is written inside the workflow workspace for packaging only; it is not committed back to the repository.

## Production Release Flow

1. Open a pull request from `develop` into `main`.
2. Merge only after CI passes and the release is approved.
3. Create the production tag from `main`:

   ```bash
   git checkout main
   git pull origin main
   git tag v1.1.0
   git push origin v1.1.0
   ```

4. The release workflow verifies that the tag points to a commit reachable from `origin/main`.
5. The workflow syncs the package-only version from the tag, builds assets, packages `creatorstack-ai.zip`, verifies required files, and publishes a GitHub Release.

Release notes are generated by GitHub and grouped by `.github/release.yml`.

## Branch Protection

Configure GitHub branch rules for `develop` and `main`:

- Require pull requests before merging.
- Require the `Run All Tests` status check.
- Require branches to be up to date before merging.
- Restrict direct pushes to maintainers.
- Do not allow force pushes or branch deletion.

For `main`, also require that production releases are created by tags. Merging to `main` should not publish a release by itself.
